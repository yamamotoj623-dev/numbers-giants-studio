> scripts の構成・連続ルール・アウトロの正典。

## 1. scripts 全体構造

### 1.1 必須ルール

| 項目 | ルール |
|---|---|
| **総 scripts 数(縦長 9:16)** | **60 秒以内に収まる範囲で柔軟**(目安 20-30、固定ではない) |
| **総 scripts 数(横長 16:9)** | 動画長に応じて柔軟(2-3 分目安 30-50) |
| **id 連番** | id 1から連番、欠番禁止 |
| **動画長** | 28-30 id × 平均1.8秒 = **50-55秒** (60秒制限内) |
| **同 speaker 連続** | **最大2回まで** (★旧 4回から強化★) |
| **レイアウト切替** | 動画内 2-3 回 (15秒間隔以上、意味ある時のみ) |

### 1.2 4フェーズ構成

```
フェーズA (フック)     id 1            isCatchy: true
フェーズB (導入)       id 2-5          視聴者代弁・テーマ提示
フェーズC (本論)       id 6-25         現象+データの深掘り、ハイライト
フェーズD (アウトロ)   id 末尾2-3      自然な締め + 二択誘引
```

#### フェーズA (id:1) - フック=動画タイトル
- ★詳細は **hook-design.md** を必ず参照★
- isCatchy: true (固定 46px の大きな文字)
- 強調記号最低2箇所
- 主語/固有名詞/インサイト/3-4行×6-8字 必須

#### フェーズB (id:2-5) - 導入
- B (もえかちゃん) が視聴者代弁で振る
- A (数原さん) が「今日のテーマ」を提示
- ★A↔B の呼び合いがここで最低1回入る★
  - 例: B「数原さん!今日の○○ですね」/ A「そうですもえかちゃん」

#### フェーズC (id:6-25) - 本論
- 現象→数字→解釈の3段階で1指標を語る
- レイアウト切替で飽きを防ぐ (2-3回)
- ハイライト機能で「今この数字を話してます」を視覚化

#### フェーズD (id:末尾2-3) - アウトロ
- 自然な締め or 文脈ある二択
- ★末尾2 scripts は「両論併記+選択を視聴者に投げる」のが理想★
- 詳細は本ファイル「3. アウトロ設計」参照

## 3. アウトロ設計 (★末尾2-3 scripts、文脈ある二択)

### 3.1 自然な締めパターン (推奨)

末尾2-3 scripts は **「両論併記+選択を視聴者に投げる」**:

```
A (倒数2): "結論。○○で△△を崩し、××がねじ伏せるか"
A (倒数1): "それとも『天敵』○○の一発に沈むか"
B (末尾):  "皆さんは○○派?△△派?"
```

→ 「勝つ」「快投」と断定せず、**両方の可能性をデータで提示してから視聴者に問う**。
これがコメント誘発+視聴維持率を両立する最強パターン。

### 3.2 NG: 唐突な二択 (廃止ルール)

```
B: "皆さんは10本派?20本派?"  (前振りなし)
```
→ 文脈なしの二択は離脱誘発。必ず**直前で両論併記**してから問う。

### 3.3 NG: 定型句

```
B: "コメントで教えてください"  ← 定型句、離脱率最悪
B: "いいねと登録お願いします"   ← 同上
```

★ 「コメントで教えて」は実投稿動画で離脱率が顕著に高かった鉄板NGパターン。

### 3.4 二択疑問の良い例集

| 動画テーマ | 二択 |
|---|---|
| 投手の試合プレビュー | 「快投を信じる派？天敵に注意派？」 |
| 打者の覚醒 | 「本物だと思う派？まだ早い派？」 |
| 監督采配 | 「采配支持派？反対派？」 |
| 編成論 | 「2軍降格派？我慢使い続ける派？」 |
| 不振選手 | 「化ける派？このまま終わる派？」 |

## 4. レイアウト切替パターン

### 4.1 動画テーマ別パターン

| テーマ | 推奨レイアウト遷移 |
|---|---|
| 朗報/覚醒 | radar_compare → timeline → ranking |
| 悲報 | radar_compare → batter_heatmap → team_context |
| 対決 (投手 vs 投手等) | versus_card → radar_compare → player_spotlight |
| チーム/采配 | team_context → ranking → player_spotlight |
| 順位/比較 | ranking → player_spotlight → versus_card |
| 投手分析 | radar_compare → pitch_arsenal → player_spotlight |
| 打者分析 | radar_compare → batter_heatmap → timeline |

### 4.2 切替ルール

- 切替最低 **15秒間隔** (5 script以上同じレイアウト)
- 切替 script に **se: "transition_swoosh"** 追加
- レイアウト切替時は **layoutType フィールド** で指定

### 4.3 パターンの2本連続使用禁止

同一パターンの2本連続使用は禁止 (動画ごとに変える)。

## 5. ハイライト継続ルール

### 5.1 ハイライト = 「今この数字を話してます」を視覚化

text 内で指標について話している期間中、`highlight: "k_9"` のように指定する。
これが続いてる間、レイアウト内の該当行/数値が強調される。

### 5.2 継続ルール

```
id:9  highlight: "k_9"      "K/9を見てください"
id:10 highlight: "k_9"      "井上は驚異の【11.20】"
id:11 highlight: "k_9"      "1試合で11個も三振"
id:12 highlight: なし        "話題が逸れたので外す"
id:13 highlight: "bb_9"     "次はBB/9を見てください"
```

★ 同じ指標の話が複数 script に渡る場合、**全ての該当 script に同じ highlight** を設定する。

### 5.3 ★v5.14.0★ versus_card / player_spotlight の行強調 (基本成績の迷子防止)

これまで `highlight` は phase==='highlight' の HighlightCard でしか視覚化されなかったが、
**v5.14.0 から versus_card / player_spotlight の通常表示でも行強調が発火**する:

- comparison.label と versus_card.categoryScores[].label が一致 → その行が脈動 + 数字拡大 + 「話題中」バッジ
- comparison.label と player_spotlight.primaryStat.label or stats[].label が一致 → 該当カードが脈動 + 「話題中」バッジ

これにより、A が「井上は驚異の【11.20】」と話してる時、画面のどこに 11.20 があるか**一目瞭然**になる。

★ Geminiは highlight をしっかり指定すること。指定が無いと行強調は発火しない。
★ ラベル一致は **完全一致 or 双方向部分一致 (case-insensitive)**。
   例: comparison.label="K/9" / categoryScores[].label="奪三振率(K/9)" でも一致する。

## 8c. ★v5.18新★ smartLoop (シームレスループ前提)

トップレベルの `smartLoop: true` (デフォルト) で、末尾 → 冒頭にシームレスループ。

このため:
- アウトロの「今日のまとめ」「チャンネル登録お願い」静止画は**表示されない**
- 末尾2-3 scripts は**会話の自然な締め**でちょうど終わるよう設計
- 視聴者は「あれ、もう2週目に入ってた」と気づかないループを目指す
- `outroCta` は出力に含めても表示されない (互換性のため受け取るが無視)
- 末尾の `se: "outro_fade"` も effective には鳴らない

逆に **smartLoop を OFF にしたい場合** (詳細結果見せたい等) は明示的に `smartLoop: false` を指定。
基本は **省略 or true でOK** = ループ前提で締めを設計する。

## 10. ★v5.20.10★ ミステリー型構成 (アハ体験の創出)

### 推奨展開順序 (チャンネル憲章)

```
① 現象 (序盤)    : 現場で起きている事象を提示
   例: 「アウトの取り方が変わった」
       「有利カウントで打たれる」
       「球速は上がってないのに被打率激減」

② 衝撃データ    : 現象の裏付けとなる数字を叩きつける
   例: 「P/IP が前年比 -1.2 球」
       「初球ストライク率 62% → 75%」
       「対左 IsoP が .045 → .172」

③ 権威の言葉    : OB/解説者/本人コメントで補強 (一次ソース)
   例: 「○○氏は『出力を抑えた配球が効いてる』と分析」
       「本人は『フォームより配球の意図を変えた』と語る」

④ 根本原因      : 終盤で「なぜそうなったか」の真因を明かす
   例: 「実は極度の打撃不振による焦りからの反動」
       「データ班の助言で配球バリエーションを増やした」
```

### NG パターン

❌ id:1 で全結論 → 視聴者「もう答え見たから離脱」
❌ id:1 で「打率の話」と言って **id:8 まで配球の話** → 期待ズレで離脱
❌ id:2 から根本原因を語る → アハ体験ゼロ、教科書朗読

### OK パターン

✅ id:1: 「奪三振を封印!?」 (ギャップ・謎)
✅ id:2-3: 状況設定 (キャラ呼び合い)
✅ id:4-7: 核心データを早めに提示 (期待値の回収)
✅ id:8-20: 多角的に深掘り (現象/データ/権威の言葉)
✅ id:21-26: 根本原因の解明 (アハ体験)
✅ id:27-30: 締め

## 11. ★v5.20.10★ テロップの文字量とテンポ

### 1 ID あたりの文字量

- **text (テロップ)**: 1 ID あたり **2-12 文字 × 最大 4 行** が読みやすい上限
- 13文字以上は2idに分割推奨 → スピーディーな展開
- 1秒以下のシーンが続くと脳が処理しきれないので、文字量と発話時間を整合させる

### 句点禁止ルール (★絶対★)

- **text には句点「。」を使わない**
  - テロップ視認は瞬間的で、句点は冗長 + 行末違和感
  - 改行 (`\n`) で意味の区切りを表現する
- speech (TTS) には句点 OK (むしろ自然な間として推奨)

### 例

❌ NG:
```json
{ "text": "今季の打率は3割。\n突出した数字です。", "speech": "今季の打率は3割。突出した数字です。" }
```

✅ OK:
```json
{ "text": "今季の打率は\n【3割】\n突出した数字", "speech": "今季の打率は3割。突出した数字です。" }
```
