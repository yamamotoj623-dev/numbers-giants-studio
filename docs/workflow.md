# 動画制作ワークフロー (Grok × Gemini × アプリ)

## 全体フロー (2方式)

### 方式A: Grok エージェント全自動 (★推奨、最速)

```
┌──────────────────────────────────────────────────────────────────┐
│ Grok エージェント (grok-agent-master.md)                          │
│  ユーザー: 「○○選手の動画作って」                                │
│  ↓ 自動実行                                                       │
│  - 旬ネタ判定 (必要なら) → リサーチ → JSON生成 → 自己レビュー    │
│  - 致命的問題は自分で修正してから出力                            │
│  ↓                                                                │
│  完成 JSON を ```json``` ブロックで出力                          │
└──────────────────────────────────────────────────────────────────┘
                          ↓ JSON を コピー
┌──────────────────────────────────────────────────────────────────┐
│ アプリ (app-v5) で JSON ペースト                                  │
│  → プレビュー確認 → 動画ダウンロード (mp4自動)                    │
└──────────────────────────────────────────────────────────────────┘
                          ↓
            YouTube Shorts 投稿 (合計約15-20分/本)
```

### 方式B: 段階的フロー (品質重視・難案件向け)

```
┌─────────────────────────────────────────────────────────────────┐
│ Step 1: ネタ発掘 (週1回)                                          │
│ Grok ③ trend_hunter → 今週の旬ネタ10本リストアップ                │
└─────────────────────────────────────────────────────────────────┘
                            ↓ ネタを1つ選択
┌─────────────────────────────────────────────────────────────────┐
│ Step 2: 詳細リサーチ (動画ごと)                                    │
│ Grok ① researcher → 選手データ・セイバー指標・直近トピック        │
└─────────────────────────────────────────────────────────────────┘
                            ↓ Markdownデータ
┌─────────────────────────────────────────────────────────────────┐
│ Step 3: JSON生成                                                   │
│ Gemini Custom Gem → schemaVersion 5.0.0 形式の JSON 出力          │
└─────────────────────────────────────────────────────────────────┘
                            ↓ JSON
┌─────────────────────────────────────────────────────────────────┐
│ Step 4: レビュー                                                   │
│ Grok ② critic → ★維持率/継続率の観点でJSON品質チェック★          │
│  - 致命的問題があれば → Step 5 へ                                  │
│  - OK なら → Step 6 へ                                             │
└─────────────────────────────────────────────────────────────────┘
                            ↓ 修正指示
┌─────────────────────────────────────────────────────────────────┐
│ Step 5: 修正 (差分更新)                                            │
│ Gemini Custom Gem → Grok ② の修正指示テキストを与えて再生成        │
│  → Step 4 に戻ってもう1度レビュー                                  │
└─────────────────────────────────────────────────────────────────┘
                            ↓ 確定 JSON
┌─────────────────────────────────────────────────────────────────┐
│ Step 6: 動画化                                                     │
│ アプリ (app-v5) で JSON 貼り付け → プレビュー確認 → mp4ダウンロード │
└─────────────────────────────────────────────────────────────────┘
                            ↓
              YouTube Shorts 投稿 (合計約25-30分/本)
```

## カスタム指示の使い分け

### Gemini (1パターン)
- Custom Gem 名: 「数字で見るG党 JSON生成器」
- 指示書: `gemini-custom-prompt.md`
- 役割: schemaVersion 5.0.0 形式の JSON を1ショットで生成
- なぜ1パターン: Gemini 3 系は冗長プロンプトを嫌う、生成は集中させた方が品質高い

### Grok の使い方 (Grok 4.3 Beta カスタムエージェント前提)

#### Grok 4.3 Beta の制約
- カスタムエージェントは **最大4個**まで作成可能
- 各エージェントの Custom Instructions は **4,000字以内**
- → 本プロジェクトの 4 つの指示書はすべて 4,000字以内に最適化済み
- 推奨契約: **SuperGrok Heavy ($300/月)** で Grok 4.3 Beta を使う

#### Workspace (プロジェクト) で運用
- Grok の Workspace 機能を使うと、プロジェクト固有の指示+ファイル添付+履歴を分離管理できる
- ワークスペース指示書: `grok-project-instructions.md` (3447字)
- ワークスペースに以下を**Knowledge ファイル**としてアップロード:
  - `yomigana-dictionary.csv`
  - `gemini-custom-prompt.md` (参考用)
  - `workflow.md` (参考用)
- ワークスペース内で4つのエージェントを呼び出す → 共通文脈+各エージェント詳細指示の両方が効く

#### 推奨セットアップ: 4枠フル活用

| 枠 | エージェント名 | 指示書 | 用途 |
|---|---|---|---|
| 1 | 数字で見るG党マスター | `grok-agent-master.md` | 全自動 (普段これだけでOK) |
| 2 | 数字で見るG党 リサーチ | `grok-1-researcher.md` | 詳細データ調査 |
| 3 | 数字で見るG党 監修者 | `grok-2-critic.md` | JSON レビュー |
| 4 | 数字で見るG党 トレンド | `grok-3-trend.md` | 旬ネタ発掘 |

各エージェントに共通で **web_search**, **x_search** ツールを有効化推奨。

#### 方式A: マスター1人で全自動 (★普段の運用)
- マスターに「○○選手の動画作って」と依頼するだけ
- 内部で 5モード自動判定 (TREND/RESEARCH/FULL_AUTO/CRITIC/EDITOR)
- 完成JSONを ```` ```json ```` ブロックで出力 → アプリにペースト

#### 方式B: 専用4枠を使い分け (難案件・品質重視)
- Trend → Research → (Geminiでjson生成) → Critic の手順で進める
- 各ステップで人間が介入可能
- 炎上リスクのある擁護動画、繊細な選手批評等で使用

## こんな時はどうする

### Q. ネタ発掘 (Step 1) を毎回やるの?
A. 週1回まとめて10本ぐらい仕込む。1日1本ペースなら週次で十分。

### Q. Step 4のレビューで OK ならStep 5は不要?
A. その通り。1回で完璧なら直接 Step 6 へ。実際は8-9割修正が入る想定。

### Q. Grok ② (critic) の指摘が厳しすぎたら?
A. 「致命的な問題」だけ修正、「改善余地」は次回以降の参考にする。完璧主義より投稿頻度重視。

### Q. 旬を逃したネタは捨てる?
A. 旬度Sは48時間で価値消滅。旬度A は1週間。定番ネタ(個人深掘り等)は旬関係なし。

### Q. Grok と Gemini どっちにリサーチさせるの?
A. **Grok**。xAI の最新データアクセスが強い。Gemini は知識カットオフが古いので最新成績は不正確。

### Q. JSON 生成を Grok にやらせるのは?
A. 非推奨。Gemini Custom Gem に最適化された指示書がある。Grok にやらせるとフォーマットずれが多発。

## YouTube投稿時のタイトル/サムネ設計

### YouTubeタイトル (動画ファイル名・投稿タイトル)
hook と同じ思想で設計する。

NG パターン (実例で視聴維持16%まで悪化したデータあり):
- ❌ 「【巨人】〜は『大正解』だった!?データが暴く驚愕の○○」
- 「だった!?」「驚愕の」「ヤバい」= 釣り動画認定 → スワイプ加速
- 「データが暴く」= 説明文、不要
- 36字超 = スマホUIで途中切れ

OK パターン:
- ✅ 「【巨人】犠打80→18 阿部監督の大正解」(短い、数字、断定)
- ✅ 「増田陸 四球ゼロ29打席の異常」(数字+感情ワード)
- 25字以内、数字を冒頭に、断定語で締める

### サムネは hook 画面そのまま使うのが基本
- アプリで生成した動画の0.5-1秒目をスクショ
- hook テキスト + stats が見える状態
